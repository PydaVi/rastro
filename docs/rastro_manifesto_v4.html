RASTRO.

O problema da segurança em cloud não é detectar risco.
É entender quais riscos se conectam em um caminho real de comprometimento — e fazer isso na mesma velocidade que um atacante moderno.

01 — Por que as abordagens atuais não bastam
pentest tradicional
Não escala com a cloud
Cloud muda diariamente. IAM é combinatório. Policies, roles e trust relationships criam explosão de caminhos possíveis que nenhum humano consegue mapear manualmente com a frequência necessária. Cada engajamento é uma fotografia de um ambiente em movimento constante.
depende de humano → não acompanha a velocidade
BAS — Breach & Attack Simulation
Testa ataques conhecidos
Breach and Attack Simulation valida se seus controles detectam ameaças já catalogadas. É útil — mas é essencialmente um teste de assinatura. Não raciocina sobre o seu ambiente específico. Não descobre que aquela combinação de permissões naquela conta abre um caminho que nenhuma assinatura conhece.
mecânico → não descobre, só confirma
CSPM / CNAPP
Mostra riscos, não caminhos
Wiz, Prisma, CrowdStrike — todos mostram findings isolados e grafos teóricos. Nenhum prova que o caminho é realmente explorável. Um finding de "role com permissões excessivas" fica na fila de remediação indefinidamente porque parece abstrato. Rastro executa o encadeamento e entrega a prova.
estático → mostra risco, não comprometimento
02 — A tese

Se ataques se tornam raciocínio automatizado, a defesa precisa se tornar raciocínio automatizado. Um agente que conhece o ambiente, formula hipóteses de comprometimento, testa cada cadeia, descarta o que não funciona, e prova o que funciona — com evidência auditável em cada passo.

Não é um scanner. Não é um playbook de ataques conhecidos.
É um engine que pensa sobre o seu ambiente específico
da mesma forma que um atacante competente pensaria.

03 — Como funciona
01
Conhece o ambiente
Enumera identidades, permissões efetivas, recursos e relações de trust. Não assume nada — descobre dinamicamente a partir do estado real da conta.
02
Formula hipóteses
Identifica candidatos a pivô, raciocina sobre qual caminho tem maior probabilidade de levar ao objetivo, e prioriza antes de agir.
03
Testa e descarta
Executa cada cadeia. Quando um branch não leva a lugar nenhum, marca como falho e tenta a alternativa — sem revisitar dead-ends, sem loop infinito.
04
Prova com evidência
Cada passo executado é logado com API call real, resultado, e raciocínio explícito. O relatório mostra exatamente o que aconteceu — não uma simulação teórica.

04 — Princípios
// 01
Prova, não lista
Um finding de "permissão excessiva" não responde à pergunta que importa. Isso é explorável de verdade? Rastro executa o encadeamento e entrega a resposta com evidência — não uma estimativa de risco.
// 02
O engine raciocina — não o LLM
O LLM sugere o próximo passo. O engine decide se é válido, executa, observa, rastreia candidatos, marca branches falhos, e faz backtracking. Nenhuma lógica de segurança depende do modelo de linguagem.
// 03
Autorização explícita é inegociável
Rastro só executa com escopo definido e autorização documentada. Sem isso, o run não inicia. Isso está no código — não apenas no README. Segurança ofensiva sem ética não é segurança.
// 04
Open source porque acesso importa
Ferramentas enterprise custam o que a maioria das organizações não pode pagar. Rastro é livre, auditável, e self-hostável. Você pode inspecionar o que está rodando no seu ambiente — não precisa confiar em uma caixa preta.
// 05
Sem vendor de LLM obrigatório
O backend de raciocínio é configurável. Ollama self-hosted é o padrão — sem internet, sem custo, sem dados de ataque saindo do seu ambiente. Para ambientes sensíveis, isso não é opcional: é arquitetura.
// 06
Especialização, não cobertura genérica
Dois vetores. Profundidade real em cada um. AWS primeiro — com maturidade completa antes de expandir. Kubernetes como segundo vetor. Rastro não tenta cobrir tudo — tenta provar attack paths reais onde a maioria das organizações está exposta.

Attack paths reais, não teóricos

produto 02 · api · saas

Attack Path
em CI/CD

Antes do deploy. No pull request. Rastro simula o ambiente AWS que será criado pelo Terraform, raciocina sobre attack paths possíveis, e bloqueia o merge se encontrar cadeia explorável — antes que chegue em produção.

// diferença para Checkov e tfsec Checkov analisa resources isolados contra regras estáticas.
Rastro raciocina sobre o estado resultante combinado
com o que já existe na conta — e prova se abre
uma cadeia de comprometimento real.
→
Gate no PR, não auditoria pós-fato
GitHub Actions, GitLab CI, qualquer pipeline. Bloqueia o merge se encontrar attack path crítico. O custo de correção no PR é zero comparado a pós-produção.
→
Simulação do estado resultante
Combina o Terraform plan com o estado atual da conta para projetar o ambiente resultante — e roda o engine de attack path sobre esse estado antes do deploy.
→
Raciocínio sobre cadeias, não resources
Uma Lambda com role permissiva + bucket com policy aberta pode criar um attack path real. O engine encontra esse encadeamento. Regra estática não encontra.
→
SaaS ou self-hosted
API hospedada com mensalidade por organização, ou self-hosted com Ollama para ambientes que não podem mandar contexto de infraestrutura para APIs externas.
